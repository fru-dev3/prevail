# Code of conduct

prevAIl is a small open-source project. The bar for being a good contributor here is the same as the bar for being a good colleague: be honest, be patient, and stay focused on the work.

## What we expect

- **Stay on the work.** Discussion is about prevAIl — the code, the threat model, the design decisions, the user experience. Keep personal opinions about people out of issues and PRs.
- **Disagree directly.** If you think a design is wrong, say so and explain why. If you think a contribution should be reverted, open a clear PR with the reasoning. Indirection wastes everyone's time.
- **Assume competence.** Most contributors here are professionals doing this in their off-hours. Read what they wrote before reacting.
- **Be precise.** "This doesn't work" is not a bug report. "This crashes on macOS 14.3 when council is ON and lens=all" is. The same applies to reviews — "this is bad" is not a review; "this allocates inside the render hot path" is.

## What we don't accept

- Personal attacks, harassment, or sustained hostility toward any contributor.
- Comments targeting any personal characteristic (background, identity, technical experience, etc.) rather than the work.
- Demanding free labor from maintainers or other contributors.
- Spam, off-topic promotion, or AI-generated low-quality contributions that haven't been reviewed by a human before submission.

## Scope

This applies to all project surfaces: GitHub issues, pull requests, discussions, security advisories, and any other space tied to the prevAIl repo.

## Reporting

If someone's behavior is making the project unsafe or unproductive to participate in:

- For a serious or sensitive matter, open a private [GitHub Security Advisory](https://github.com/fru-dev3/prevail/security/advisories/new). It's the right channel for anything that shouldn't be aired in public.
- For lower-stakes issues, contact the maintainer directly via the email in the latest commit metadata, or open a discussion if it's about a project norm rather than a specific person.

Reports stay private. The maintainer will acknowledge within 7 days, investigate, and decide on a response — which may range from a private note to revoking access to the repository.

## Enforcement

Decisions are at the maintainer's discretion. There is no committee, no appeal process, and no formal procedure. This is a small project and we will be reasonable.

If the maintainer's own behavior is the problem, GitHub itself is the escalation path — report via [GitHub's abuse channel](https://github.com/contact/report-abuse).
